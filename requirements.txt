# -- CLIP (pre-OCR language detection in language_detector.py) ------------------
# Uses open_clip_torch, NOT the original openai/CLIP package.
# openai/CLIP's clip.load() calls torch.jit.load() then walks private
# TorchScript internals (RecursiveScriptModule._construct) that aren't part
# of torch's public API and get renamed/removed across versions without
# warning. On torch==2.6.0 this fails with:
#   AttributeError: type object 'RecursiveScriptModule' has no attribute '_construct'
# open_clip_torch is the actively maintained successor, loads the identical
# OpenAI-published ViT-B/32 weights (pretrained="openai") through a plain
# nn.Module state_dict path instead of torch.jit.load, and isn't exposed to
# this class of breakage. Pin >=2.24 to avoid the old protobuf<4 ceiling
# that 2.20.x carried (this project already needs protobuf>=4.21.6).

# -- Surya OCR ------------------------------------------------------------------
# 0.13.1 is the last version that supports torch>=2.5.1 (compatible with torch==2.6.0)
# To use the latest surya-ocr 0.20.0, upgrade torch to 2.7.0 in both places

# surya-ocr==0.13.1's own pyproject.toml requires transformers>=4.41.0,<5.0.0.
# Nothing else in this file pinned transformers directly, so a later
# `pip install -U transformers` (or a transitive pull from sentence-
# transformers / another package) can silently land on a newer 4.5x/4.57.x
# release, which breaks surya's SuryaOCRConfig with:
#   KeyError: 'encoder'  (or, on some versions: AttributeError on pad_token_id)
# raised deep inside transformers.configuration_utils.PretrainedConfig.
# This is a KNOWN recurring issue, not a one-off: the surya project itself
# has shipped hotfixes for transformers 4.56.0 compatibility before, and a
# separate open issue on their repo reports the same class of break on
# 4.57.6, fixed by downgrading to 4.57.3. surya 0.13.1 predates both of
# those fixes, so pin narrower than surya's own (too permissive) range:
transformers==4.46.3
